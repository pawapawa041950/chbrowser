# ChBrowser 設計書

5ちゃんねる(5ch.io)専用 Windows ネイティブブラウザの設計書。

- 最終更新: 2026-05-05
- ステータス: 実装中 (Phase 11 ~ Phase 23 完了、ロードマップ外で多数の周辺機能を追加実装)

## 1. 概要

| 項目 | 内容 |
|------|------|
| 目的 | Windows ネイティブな見た目の 5ch.io 専用ブラウザ |
| 対象OS | Windows 11 のみ |
| プラットフォーム | WPF (.NET 8) / C# |

## 2. 機能要件

### 2.1 閲覧
- 板一覧 / スレ一覧 / スレ本文の 3 ペイン構成
- 画像/動画のインラインサムネイル表示
- リッチスクロールバー(マルチトラック方式)
- AA(アスキーアート)常時対応(MS P ゴシック固定)
- スレ内検索(Ctrl+F、ヒット位置をスクロールバーに表示)
- 同一 ID の集計表示(`ID:xxx [3/15]`)、ID クリックで同 ID レスのフィルタ
- アンカーポップアップ(多段、たどれる)
- アンカー先レスの直下に返信レスを連結表示(chmate 方式)

### 2.2 投稿
- レス書き込み
- 新規スレ立て
- メール欄 sage チェックボックス
- 浪人/プレミアム機能は不要
- どんぐり Cookie の自動管理
- 書き込みエラー時はダイアログ表示+本文保持で再投稿可

### 2.3 管理
- NG ワード / NG 名前 / NG ID / NG ワッチョイ
- マッチ方式: 通常文字列 (部分一致) / 正規表現
- NG スコープ: グローバル / 板単位
- ルール単位の有効/無効トグル + 有効期限 (ルールごとに「無期限」または「YYYY-MM-DD まで」)
- 連鎖あぼーん (NG レスにアンカーしているレスも非表示、無限再帰)
- あぼーん表示モード: 完全非表示 (透明あぼーん) 固定 — 「あぼーん」マーカーすら出さない
- ステータスバーにあぼーん件数表示 (現在表示中のスレ単位)
- 既読位置のセッションまたぎ永続化
- お気に入り板/スレ
- マルチタブ (再起動時に復元)
- ウィンドウ位置/サイズ/ペイン幅の永続化

### 2.4 画像/動画
- 取得: viewport 近接時に HEAD でサイズ確認し、しきい値以下なら自動取得
  (lazy 読み込み = 採用。設計書当初は不採用としていたが、HEAD でサイズが分かれば帯域コストの心配がないため採用に変更)
- 5MB 超(設定可、現状固定)はクリックされるまで未取得
- 画像はインライン静止表示
  - **gif / webp**: ブラウザ既定の auto-animate に任せる。スレ閲覧では自動再生される動きが期待される挙動なので、当初設計の「クリックで再生」は採用しない
  - **直リンク動画 (.mp4 .webm .mov)**: 320×180 のスロットを描画。viewport 近接で `<video preload="metadata" muted>` を materialize し `loadedmetadata` 時に 0.1s シークして first frame をサムネ代わりに表示。クリックで `controls + unmute + play()` に切り替え (CORS / 破損で frame が出ない場合は黒コマで fallback)
  - **YouTube**: サムネイル (i.ytimg.com) を表示するのみ。クリックでデフォルトブラウザに開く (インライン iframe 再生は Error 153 が安定して出るため一旦見送り、Phase 10 のビューアウィンドウで対応を検討)
- 画像クリック → ビューアウィンドウ(別ウィンドウ)で拡大
- ビューアウィンドウ自体が複数画像をタブで保持
- 外部画像サービス(imgur, x.com, pixiv 等)の URL は実体画像 URL に展開して取得

### 2.5 キャッシュ
- 画像はハッシュ分散保存
- 上限 1GB(設定可)、超過時は LRU で自動削除

### 2.6 HiDPI
- DPI Awareness を切替可
- デフォルト: スケールしない設定(`unaware`)
- 設定で `Per-Monitor V2` に変更可(再起動要)

## 3. 5ch.io 通信仕様

### 3.1 取得エンドポイント

5ch.io エコシステムは内部に多数のサブドメイン host を持つ(`hayabusa9.5ch.io`, `mao.5ch.io`, `kizuna.5ch.io` 等、約 130 種)。各板はそのうちのいずれかにホストされる。**実際の URL は bbsmenu.json の `url` フィールドが正**。

| 用途 | URL | 形式 | エンコーディング |
|------|-----|------|----------------|
| 板一覧 | `https://menu.5ch.io/bbsmenu.json` | JSON | UTF-8 |
| 板設定 | `<board.url>SETTING.TXT` | テキスト | Shift_JIS |
| スレ一覧 | `<board.url>subject.txt` | テキスト | Shift_JIS |
| dat | `<board.url>dat/<key>.dat` | dat | Shift_JIS |
| read.cgi(参考) | `https://<host>/test/read.cgi/<dir>/<key>/` | HTML | UTF-8 |

例: ニュース速報板は `https://hayabusa9.5ch.io/news/` がベース URL なので、subject.txt は `https://hayabusa9.5ch.io/news/subject.txt`。

- スレ本文の取得は **dat 形式を使用**(read.cgi HTML は使わない)
- 取得はすべて HTTPS
- User-Agent: `Monazilla/1.00 ChBrowser/<version>`(Monazilla 規約準拠)
- 一部エントリ(`twitter.com`, `5ch.works` 等)は実板ではなくナビゲーションリンクなので除外する

### 3.2 認証(B11 仕様)

5ch.io 移行に伴い旧 Monazilla API(AppKey + HMKey による HMAC 署名)は廃止された。

- `X-MonaKey` / `X-PostSig` / `X-APIKey` / `X-PostNonce` の各ヘッダは **送信しない**
- AppKey/HMKey の開発者登録は **不要**
- 認証は Cookie で行う:
  - `MonaTicket=...`(旧 `yuki=akari` の置換、書き込み認証)
  - `acorn`(どんぐり Cookie)
- 詳細パラメータ(`MonaTicket` の発行条件等)は実装時に [JDim](https://github.com/JDimproved/JDim) v0.16.0 以降と [2chAPIProxy](https://github.com/onihusube/2chAPIProxy) のソースを参照して確定

### 3.3 差分取得(Range)

スレ再読み込み時は HTTP `Range: bytes=N-` で末尾差分のみ取得する。

- N = 既保存 dat のバイト長
- レスポンスが 206 Partial Content なら追記、200 なら全置換
- 416 Range Not Satisfiable はスレが縮んだ(あぼーん編集等)場合 → 全取得し直し
- 自動更新は無し、手動リロード時のみ動作

**ディスクファースト表示** (UX 上の重要ポイント): ローカルにキャッシュ済 dat があるスレを開く時、`OpenThreadAsync` は HTTP リクエストを投げる前に `LoadFromDiskAsync` でディスクから dat を読み・パースして即表示する。HTTP の差分取得 (Range リクエスト → サーバ応答 → 新着分のパース) はそのあと裏で走り、新規分のみ末尾に append される。これにより既読スレの再オープンが「HTTP roundtrip 待ち (200ms〜1秒)」抜きで体感的に瞬時になる。

### 3.4 書き込み・スレ立て

- POST `https://<board>.5ch.io/test/bbs.cgi`
- Content-Type: `application/x-www-form-urlencoded`
- ボディ: Shift_JIS で URL エンコード
- 主要フィールド:

| フィールド | 内容 |
|----------|------|
| `bbs` | 板ディレクトリ名(例: `news`) |
| `key` | スレッドキー(レス時のみ) |
| `subject` | スレタイトル(スレ立て時のみ) |
| `time` | エポック秒 |
| `FROM` | 名前 |
| `mail` | メール欄(`sage` 等) |
| `MESSAGE` | 本文 |
| `submit` | `書き込む` または `新規スレッド作成` |

- 2 段階フロー:
  1. 1 回目 POST → サーバーが確認画面 HTML を返す(クッキー Set-Cookie 付き)
  2. 同じ URL に同じパラメータで再度 POST(Cookie を付与)→ 投稿確定
- エラー時(規制/レベル不足/`broken_acorn` 等)はダイアログ表示。本文を保持し再投稿可能にする。

### 3.5 どんぐり(acorn)

| 項目 | 内容 |
|------|------|
| Cookie 名 | `acorn` |
| Domain | `.5ch.io`(推定) |
| 発行 | 初回書き込み時に Set-Cookie |
| レベル上昇 | 時間経過(0→1 で約 5 分) |
| 有効期限 | 約 3 時間で「枯れる」(失効) |
| IP 紐付き | 接続変更で Lv0 にリセット |
| Lv 表示 | 名前欄に `!donguri` を入力すると確認可 |
| 復旧 | `broken_acorn` エラー時は Cookie を削除して再構築 |

レベル不足エラーは投稿ダイアログに表示する。レベル UP のリトライ予測時刻を出すと親切。

**将来検討 (Phase 8 では未実装)**: 5ch には Cookie + 時間経過とは別に **メールアドレス認証どんぐり** (= 認証済どんぐり) が存在し、メアド確認による初期 Lv 引き上げや有効期限延長の経路がある。Phase 8 ではまず Cookie + 時間経過版を通し、メアド認証どんぐりは後続フェーズで追加する。

### 3.6 規制

- 国外 IP は HTTP 451
- 連投制限の具体秒数は不明 → 実装時に挙動から確認
- ステータスバーまたは投稿ダイアログにエラー詳細を表示

## 4. データ永続化

### 4.1 ディレクトリ構造

**ポータブルアプリとして配置する** — `ChBrowser.exe` と同じディレクトリ配下の `data/` に全データを置き、その exe ディレクトリごと別マシンに持っていけばそのまま動作するようにする。`%LOCALAPPDATA%` 等のユーザプロファイル配下は使わない。

```
<exe ディレクトリ>/
  ChBrowser.exe
  ChBrowser.dll
  ...
  data/
    app/
      config.json          ; アプリ全体設定
      layout.json          ; ウィンドウ位置・サイズ・ペイン幅 (タブ復元は今後の検討事項)
      favorites.json       ; お気に入り(スレ/板、フォルダ階層含む)
      shortcuts.json       ; ショートカット / マウス操作 / マウスジェスチャーのバインド (Phase 15)
    ng/
      rules.json           ; 全 NG ルール (グローバル + 板単位を 1 ファイルに統合、Phase 13e)
                           ; 旧 global.json + by_board/<board>.json は NgStorage.LoadAndMigrate が起動時に自動移行 + 削除
    donguri/
      cookies.txt          ; どんぐり Cookie 値(Netscape 形式)
      state.json           ; 最終取得時刻、推定 Lv 等
    cache/
      images/
        <hh>/<hash>.<ext>  ; ハッシュ分散(SHA256 の先頭 2 hex)
        index.json         ; ハッシュ→元 URL/最終アクセス/サイズ
    themes/
      <theme-name>/
        post.html          ; レス 1 件分のテンプレ (5.3.1 参照)
        post.css
    5ch.io/
      bbsmenu.json         ; 板一覧キャッシュ (menu.5ch.io 由来)
      <directory_name>/    ; 例: news, livejupiter (host は問わずここに集約)
        _board.json        ; SETTING.TXT 由来メタ + 実 host
        _subject.txt       ; subject.txt 原本(Shift_JIS)
        <key>.dat          ; dat 原本(Shift_JIS)
        <key>.idx.json     ; メタ: 既読位置、被アンカー集計、URL/画像/動画位置
    bbspink.com/
      <directory_name>/
        ...
```

実 host は多数あるが、root domain (`5ch.io` / `bbspink.com`) ごとに集約し、その下に `directory_name` で板ディレクトリを切る。`directory_name` は同一 root domain 内では一意であることを bbsmenu.json で確認済(2026/04 時点)。

**実装メモ** (`Services/Storage/DataPaths`):
- 既定の `Root` は `Path.Combine(AppContext.BaseDirectory, "data")` (= exe と同階層の `data/`)
- 開発時に別ディレクトリを使いたい場合は環境変数 `CHBROWSER_DATA_DIR` で上書き可能
- 旧設計 (`%LOCALAPPDATA%\ChBrowser\data` 配下) は**廃止**。既存ユーザがいた場合は移行スクリプト等で `data/` を持ってくる想定

### 4.2 ファイル形式

- **dat / subject.txt**: 取得そのままの Shift_JIS で保存(変換しない)
- **bbsmenu.json**: 取得そのまま UTF-8 で保存
- **メタ JSON / 設定 JSON**: UTF-8、`System.Text.Json` で読み書き
- **favorites.txt / cookies.txt**: UTF-8 テキスト
- すべて人間がエディタで読める形式

### 4.3 dat 形式

5ch 標準形式(1 投稿 1 行):
```
名前<>メール<>日付 ID<>本文<>タイトル(1 レス目のみ)
```
- 本文中の改行は ` <br> ` で表現
- 改行コードは LF(`\n`)
- SJIS にない文字は HTML 数値文字参照(`&#xXXXX;`)

### 4.4 idx.json 構造

現在実装している最小構造 (将来 NG キャッシュなどを段階的に追加予定):

```json
{
  "lastReadPostNumber": 152,
  "lastFetchedPostCount": 287,
  "lastReadMarkPostNumber": 148,
  "ownPostNumbers": [12, 47, 98]
}
```

| フィールド | 用途 |
|-----------|------|
| `lastReadPostNumber` | 最後にビューポート上端にあったレス番号。次回オープン時の自動スクロール復元に使う。 |
| `lastFetchedPostCount` | 前回 dat 取得完了時のレス総数。subject.txt の現在件数と比較して新着の有無を判定 (スレ一覧の青丸 / 緑丸 マーク)。 |
| `lastReadMarkPostNumber` (Phase 19) | 「ここまで読んだ」帯の対象レス番号。primary レス (= ツリーの子表示は除外) の最深既読位置を JS が追跡し、増加方向のみ更新 (= 値は減少しない)。dat 削除で idx.json ごと消えるため自動リセットされる。 |
| `ownPostNumbers` | 「自分の書き込み」としてマークされたレス番号配列。スレ表示のレス番号メニュー (5.3 参照) で個別にトグル可能。null / 未定義は空集合と同義。dat 削除で idx.json ごと消えるため自動リセットされる。 |

将来追加予定 (各フェーズでの拡張):

| 追加予定フィールド | 由来フェーズ | 用途 |
|---|---|---|
| `anchorCount: { "N": count }` | Phase 5 拡張 | 被アンカー数キャッシュ。現状は表示のたびに JS で都度計算 (`buildReverseIndex`) |
| `links: { "N": { url, image, video } }` | Phase 6 | 画像/動画マーカー、URL 展開キャッシュ |
| `ngHidden: [N, ...]` | Phase 13 | あぼーん集計のキャッシュ |

### 4.4.1 layout.json 構造

`data/app/layout.json` にウィンドウ位置/サイズと主要ペインの寸法を保存する。アプリ終了時 (`MainWindow.OnClosing`) に書き出し、起動時 (`MainWindow.OnSourceInitialized`、Show 直前) に読んで適用する。

```json
{
  "windowLeft": 100,
  "windowTop": 50,
  "windowWidth": 1200,
  "windowHeight": 720,
  "windowMaximized": false,
  "boardListWidth": 200,
  "threadListHeight": 240
}
```

| フィールド | 用途 |
|---|---|
| `windowLeft` / `windowTop` / `windowWidth` / `windowHeight` | ウィンドウの位置とサイズ。`windowMaximized=true` の時は `RestoreBounds` (= 通常表示時の値) を保存している |
| `windowMaximized` | 最大化状態。次回起動時はまず通常サイズで復元してから最大化する |
| `boardListWidth` | 左カラム (お気に入り + 板一覧) の幅。XAML 上は `ColBoardList` |
| `threadListHeight` | 右上ペイン (スレ一覧) の高さ。XAML 上は `RowThreadList` |
| `favoritesPaneHeight` | 左カラム上の **お気に入りペイン** の高さ。XAML 上は `RowFavorites` (Phase 7 で追加) |

`ColRight` (右ペイン全体) や `RowThreadView` (右下スレ表示) は Star サイズ (`*`) で残スペースを自動的に吸うため明示保存しない。

書き込みは `.tmp` に書いてから rename して atomic 化する (途中 kill された時の壊れた JSON で起動できなくなるのを防ぐ)。読み込み失敗 (ファイル不在 / 破損 / 値が NaN) は無視して XAML のデフォルト値で起動する。

タブ復元 (どのスレ/板を開いていたか) は今後の検討事項 — 着手時に `layout.json` を拡張するか別ファイル化するか決める。

### 4.4.2 favorites.json 構造

`data/app/favorites.json` にお気に入り (板/スレ + フォルダ階層) を保存する。`FavoritesStorage` が atomic 書き込み (`.tmp` → rename)。Phase 17 で UI 上の「お気に入り」仮想ルートフォルダは表示専用 (5.4.1 参照) で、永続化対象には含まれない。

```json
{
  "version": 1,
  "root": [
    {
      "type": "folder",
      "id": "61f4e2c0-0000-0000-0000-000000000001",
      "name": "ニュース系",
      "addedAt": "2026-04-27T10:00:00+00:00",
      "children": [
        {
          "type": "board",
          "id": "...",
          "host": "hayabusa9.5ch.io",
          "directoryName": "news",
          "boardName": "ニュース速報",
          "addedAt": "2026-04-27T10:01:00+00:00"
        },
        {
          "type": "thread",
          "id": "...",
          "host": "hayabusa9.5ch.io",
          "directoryName": "news",
          "threadKey": "1700000000",
          "title": "雑談スレ★123",
          "addedAt": "2026-04-27T10:02:00+00:00"
        }
      ]
    },
    {
      "type": "thread",
      "id": "...",
      "host": "...", "directoryName": "...", "threadKey": "...", "title": "...",
      "addedAt": "..."
    }
  ]
}
```

| 共通フィールド | 用途 |
|---|---|
| `type` | `folder` / `board` / `thread` の判別子 (`System.Text.Json` の polymorphic discriminator) |
| `id` | GUID。ユーザー操作 (移動・削除) で対象を特定するためのキー |
| `addedAt` | 追加日時 (DateTimeOffset、UTC)。デフォルトの並びは降順 (新しいものが上) |

| folder 固有 | 用途 |
|---|---|
| `name` | フォルダ表示名 |
| `children` | 子エントリの配列 (folder/board/thread を任意深さでネスト可) |

| board 固有 | 用途 |
|---|---|
| `host` / `directoryName` / `boardName` | 板情報。bbsmenu.json 失われても表示できるよう boardName も保存 |

| thread 固有 | 用途 |
|---|---|
| `host` / `directoryName` / `threadKey` / `title` | スレ識別 + タイトル (subject.txt から消えても表示できるよう保存) |

### 4.5 画像キャッシュ

- ハッシュ分散: `images/<hash[0:2]>/<hash>.<ext>`
  - hash = `SHA256(url)` の hex 全 64 文字 (URL ベースなので 1 URL = 1 ファイル、内容ハッシュではない)
  - ext = URL 拡張子優先。無ければ Content-Type から推測 (`image/jpeg` → `.jpg` など)、最終フォールバック `.bin`
- 書き込みは `<hash>.<ext>.tmp` → rename で atomic 化 (書き込み中の読み取り防止)
- `images/index.json`: snake_case JSON、key = hash
  ```json
  {
    "<hash>": {
      "hash": "<hash>",
      "url": "https://i.imgur.com/abc.jpg",
      "ext": ".jpg",
      "size": 234567,
      "content_type": "image/jpeg",
      "fetched_at": "2026-04-25T10:00:00+00:00",
      "last_access_at": "2026-04-25T11:00:00+00:00"
    }
  }
  ```
- 上限 1GB (`MaxBytes`、Phase 11 で設定可能化予定)。超過時は `last_access_at` 昇順で LRU 削除
- 1 ファイル上限 64MB (`MaxFileBytes`)。これを超えるレスポンスはキャッシュしない (RAM/IO 保護)
- `last_access_at` の更新はメモリ上で行い (`_dirty` フラグ)、`SaveAsync` / `PruneIfOverLimitAsync` / `Dispose` 時に index.json を一括 flush

## 5. UI 設計

### 5.1 メインウィンドウ全体

```
┌───────────────────────────────────────────────────────────────┐

### 5.2.1 アドレスバー (Phase 14)

メニューバーと同じ行の右側に配置する単一行 TextBox。Chrome / Edge のアドレスバーと同じ感覚で、現在開いている板/スレの URL を表示し、URL を貼って Enter で開く入口になる。

**配置**:
- メニューバー行を `DockPanel` でラップし、`Menu` を Left dock (内容分の幅)、`TextBox` が残り全幅を占める (= LastChildFill)
- 簡易代替案として「メニューの 1 段下に独立した行」もあり得るが、本実装では同一行配置を採る
- ウィンドウ幅が極端に狭い場合は TextBox 側が圧縮される (Menu は内容幅を確保)

**表示する URL の決定** (last-activated wins):
- 右ペインの `SelectedThreadListTab` (= スレ一覧タブ) と `SelectedThreadTab` (= スレタブ) のうち、**最後にユーザがアクティブにした方**の URL を表示
  - スレ一覧タブをアクティブ化 → `https://<host>/<dir>/`
  - スレタブをアクティブ化 → `https://<host>/test/read.cgi/<dir>/<key>/`
- アクティブ判定は両方の `OnSelectedXxxChanged` で記録 (= どちらが最後に発火したか)
- 左ペイン (お気に入り / 板一覧) はタブ概念がないためアドレスバー表示に影響しない (= 板項目をクリックして右ペインに開いた瞬間に `SelectedThreadListTabChanged` 経由で更新される)
- 表示中タブが閉じた / どちらも未選択 → 空欄
- `ThreadListTabViewModel.Board` が null の場合 (= お気に入りディレクトリ展開タブ) は空欄
- 表示する URL 文字列は `tab.Board.Url` (bbsmenu 由来の正規 URL、サブドメイン込み) をそのまま使う (= コピペ用途で他ブラウザにそのまま貼れる形)

**入力時の挙動**:
- 入力解析の詳細は §7.7 参照
- Kind=Board → 該当板のスレ一覧タブを開く (既存タブがあればアクティブ化)
- Kind=Thread → 該当スレのスレタブを開く (既存タブがあればアクティブ化)
- 既存タブとの照合キーは **(root domain, dir)** または **(root domain, dir, key)** — Phase 13 NG スコープと同じ `DataPaths.ExtractRootDomain` を流用 (= サブドメイン違いの URL を貼っても既存タブにヒット)
- 5ch.net host のみ自動的に 5ch.io に書き換えてアクセス
- 不正な URL / 5ch.io / bbspink.com 以外のホスト → アドレスバーを赤枠化 + ステータスバーへエラーメッセージ ("認識できない URL です" 等)、アドレスバーは入力テキストを保持したまま

**入力中のテキスト保持**:
- TextBox にフォーカスがある間 (= ユーザ入力中) は、上記「表示する URL の決定」による自動更新を抑止
- フォーカスが「板一覧 WebView (左ペイン)」「スレ一覧タブ WebView」「スレタブ WebView」のいずれかに移った瞬間に、現在のアクティブタブの URL に戻す
- メニュー / 設定ウィンドウ / 他のボタンへフォーカスが移っただけでは入力テキストを保持し続ける

**フォーカス補助**:
- `Ctrl+L` でアドレスバーへフォーカス + 全選択 (ブラウザ標準のショートカット)
- TextBox がフォーカス取得した瞬間にも全選択 (クリック / Tab 移動 / Ctrl+L のいずれでも上書き入力可能に)

**右クリックメニュー**:
- 標準の コピー / 切り取り / 貼り付け / 全選択 に加えて、
- 「貼り付けて移動」 — クリップボードの URL を貼り付けて即 Enter 動作

**末尾スラッシュ**:
- 板 URL は `/<dir>/`、スレ URL は `/test/read.cgi/<dir>/<key>/` を正規形として表示
- 入力時はスラッシュの有無を問わず受け付け、内部で正規化

### 5.3 スレ本文ビュー

**実装方針**: スレ本文の描画は WPF ItemsControl ではなく、**各タブが専属で持つ WebView2** + 埋め込み HTML/CSS/JS による renderer に委ねる。WPF レイヤーが受け持つのはタブ管理 / 上のツールバー / 板/スレ一覧との連携のみ。

**テーマ機構** (5.3.1): 1 レス分の HTML 構造とスタイルは **テーマファイル** (`post.html` テンプレ + `post.css` スタイル) に切り出してあり、ユーザーがテキストエディタで編集可能。詳細は 5.3.1 を参照。

レス 1 件は HTML の `<div class="post" id="rN">` で、ヘッダ (No・名前・日時・ID) と本文 (`<div class="post-body">`) を持つ。本文中の `>>N` `>>N-M` はアンカー化、`<a>` は href なしで生成 (focusable にしないため、フォーカス由来 auto-scroll を防止)。`http(s)` URL は `<a data-url>`、5ch dat 由来の `<a href="../test/read.cgi/...">>>1</a>` は post anchor として再解釈。

**表示モード** (右上ツールバーで切替、`ThreadViewMode` enum 同期):

| モード | 内容 |
|------|-----|
| flat (レス順) | 1〜N 件を順に並べるだけ。インライン展開なし |
| tree (重複あり) | 各レス直下に `>>X 参照先 (親)` と `自分を参照しているレス (子)` を 1 段だけインライン展開 |
| dedupTree (重複なし) | tree と同じ親/子展開を再帰的に行うが、各レスは正確に 1 度だけ描画 |

**多段アンカーポップアップ**: anchor をホバーすると参照先レスを近接にポップアップ。ポップアップ内の anchor をさらにホバーで多段ポップアップ。各ポップアップ独立タイマで close 制御 (上位/下位ポップアップは独立)。

**レス番号クリックメニュー**: `.post-no` (= `<span data-number="N" role="button">`) をクリックするとレス番号下にポップアップメニューを表示。スレ末端での表示は viewport 下端を超えるならアンカー上にフリップする。

| 項目 | 動作 |
|---|---|
| 返信 | 投稿ダイアログを `>>N\n` 入りで開く (`MainViewModel.OpenReplyDialog`、`PostFormViewModel.Message` プリフィル) |
| 自分の書き込みに登録 / 解除 | レス番号を `idx.json.ownPostNumbers` に追加/削除し、レンダ後の「自分」バッジ表示を toggle (詳細は次項) |
| NG (展開) | クリックでインライン子展開。子は **名前 / ID / ワッチョイ** の 3 種で、各々抽出済の値を表示 (空なら disabled)。選択で `NgQuickAddDialog` (5.3.2) を開く |

メニュー DOM は body 直下の単一 `<div class="post-no-menu">` で管理。外側 mousedown / Escape / 別レス番号クリックで閉じる。

**「自分の書き込み」バッジ**: `.post-name` の直前に `<span class="post-own">自分</span>` (青背景 + 白文字) を出力。永続化は `idx.json.ownPostNumbers`、JS 側は `appendPosts` ペイロードに同梱される `ownPostNumbers` で初期化、メニューでのトグル時は `updateOwnPosts` 増分メッセージで DOM を直接書き換え (= 全再レンダはしない)。バッジは post.html テンプレ + post.css で定義されておりユーザがテーマで装飾を変更可。

**ツールバー** (スレ一覧とスレ表示の間):
- リフレッシュ (Range で差分取得 → JS appendPosts で末尾に増分追加)
- 表示モード切替 (3 状態トグル)
- ログ削除 (.dat と .idx.json を削除してタブも閉じる)

**フォント**: `MS P ゴシック` 固定 (AA 対応)。色絵文字は WebView2 (Chromium) が ネイティブ対応。

**streaming 描画**: 初回オープンは dat を 10 → 50 件単位で C# 側がバッチして JS の `appendPosts` を逐次呼ぶ (体感即時表示)。

**スクロール位置永続化**: ユーザーがスクロールしたら 200ms debounce で「ビューポート上端のレス番号」を C# に通知、`idx.json.lastReadPostNumber` に保存。次回オープン時に自動復元。

#### 5.3.1 テーマ機構 (post.html / post.css)

1 レス分の HTML 構造とスタイルを `Resources/post.html` (テンプレ) と `Resources/post.css` (スタイル) に切り出し、ユーザー側で自由に変更できるようにしている。

**ディスクレイアウト**:
```
<exe ディレクトリ>/data/themes/
  default/
    post.html      ; テンプレ (デフォルトでは無い。Phase 11 設定で明示生成 or ユーザーが手動配置)
    post.css       ; レス本体のスタイル (.post / .post-no / .post.embedded など)
```

ロジック:
- アプリ起動時に既定テンプレを **ディスクへ自動展開はしない**。`ThemeService.LoadActiveTheme` がディスクにファイルが無ければ埋め込みリソースを読む
- **DEBUG ビルドでは disk を一切読まず常に埋め込みリソースを返す**。開発中はソース (`Resources/post.html` / `post.css`) の変更がビルド + 起動で即反映され、過去に extract された古いディスクコピーで隠されることがない
- RELEASE ビルドではディスク優先 → 埋め込み fallback の従来動作。ユーザーが上記ディスクパスに置いたカスタムは優先され、アプリ更新で埋め込み既定が変わっても保護される
- ディスクへ既定テンプレを書き出したい時は `ThemeService.ExtractDefaultThemeFiles()` を呼ぶ — Phase 11 設定画面の「テーマファイルを生成」ボタン経由

**Phase 11 (設定画面) で追加予定**:
- 「テーマファイルを生成」ボタン: `ThemeService.ExtractDefaultThemeFiles()` を呼んで現在の埋め込み既定テンプレをディスクに書き出す。既存ファイルは保護 (上書きしない)。完了後にエクスプローラでフォルダを開いてあげると親切
- 「テーマフォルダを開く」ボタン: `<exe ディレクトリ>/data/themes/default/` を Explorer で開くだけ

**Phase 12 (テーマテンプレート再読み込み) で追加予定**:
- 「テンプレートを再読み込み」ボタン: アプリを再起動せずにディスク上の `post.html` / `post.css` を読み直し、開いている全タブのスレ表示シェルを更新する
- Phase 11 で設定画面が出来ていれば、その「表示」タブにボタンを置くのが自然 (もしくはツールメニュー配下でも可)
- 実装時に必要なこと:
  - `WebView2Helper.LoadThreadShellHtml` の `_shellHtmlCache` を破棄するメソッドを生やす
  - 開いている `ThreadTabViewModel` 各々に再ナビゲーションを促す (`AppendBatch` を再 push、もしくは `NavigateToString` を再実行)。スクロール位置は `idx.json.lastReadPostNumber` から復元できるので失わない
  - ThemeService 側はキャッシュなしで毎回 `LoadActiveTheme()` するだけなので変更不要

**テンプレ言語** (Mustache 風の最小サブセット):
- `{{var}}` … 変数展開
- `{{#if var}}...{{/if}}` … truthy なら展開 (1 段ネストのみ。`null/false/0/空文字列` を falsy 扱い)
- `<!-- ... -->` … HTML コメント。**内部の `{{...}}` も含めて opaque な text として扱われる** (パーサが解釈しないので、説明文として `{{#if foo}}` を書いても安全)。出力にはそのままコメントとして残るのでブラウザは無視する

**変数一覧**:

| 変数 | 型 | エスケープ | 内容 |
|---|---|---|---|
| `number`     | int  | escape | レス番号 |
| `name`       | HTML | **raw** | 名前 (`<b>` `<small>` 等は JS 側で既処理済み) |
| `date`       | str  | escape | 投稿日時 |
| `id`         | str  | escape | 投稿者 ID。空のときは `{{#if id}}` ブロックを出さない |
| `body`       | HTML | **raw** | 本文 (アンカー含む。画像/動画スロットは `media` に分離されているのでここには含まれない) |
| `media`      | HTML | **raw** | 本文中の画像/動画/YouTube スロットを集めた HTML (`post-body` 下部に横並びで出すため `body` から分離。Jane Xeno 風) |
| `replyCount`     | int  | escape | 自分宛の返信件数。0 のとき `{{#if replyCount}}` をスキップ |
| `replyNumbers`   | str  | escape | 返信元レス番号のカンマ区切り (例: `"3,7,12"`)。バッジの `data-replies` 用 (ホバーポップアップで使う) |
| `hasFewReplies`  | bool | —      | 返信 1-2 件のとき true。`post-no` をピンクにするのに使う |
| `hasManyReplies` | bool | —      | 返信 3 件以上のとき true。`post-no` を赤にするのに使う |
| `isOwn`          | bool | —      | 「自分の書き込み」としてマークされたレスのとき true (= `idx.json.ownPostNumbers` を JS の Set に流し込み判定)。`{{#if isOwn}}<span class="post-own">自分</span>{{/if}}` を `post-name` の直前に出すのが既定 |
| `children`       | HTML | **raw** | ツリー表示で直下に展開する子レス。flat モードでは空文字 |
| `isEmbedded`     | bool | —      | 埋め込みレスかどうか (`{{#if isEmbedded}}` で `.embedded` クラス付加に使う) |
| `domId`          | bool | —      | `id="r{{number}}"` 属性を付けるか。重複 ID を避けるため tree (重複あり) モードの埋め込みは false |

**必須コントラクト** (これを破ると JS のクリック/スクロール処理が壊れる):
- 最外要素に class `post` がある
- `{{#if domId}}` 経由で id="r{{number}}" を付ける
- `{{body}}` を 1 箇所に raw 挿入する (中の `<a class="anchor">` に JS イベントが取り付く)。`{{media}}` も 1 箇所 (推奨は `<div class="post-media">{{media}}</div>` で post-body の下) に raw 挿入する。テンプレが `{{media}}` を出さないと本文中に出てきた画像/動画スロットがどこにも表示されない
- 返信件数バッジは `<span class="post-reply-count" data-replies="{{replyNumbers}}">{{#if replyCount}}...件{{/if}}</span>` の形で **常に DOM に出す** (span 自体を `{{#if replyCount}}` で囲まない、中身だけ if で囲む)。理由: streaming 中に新着レスが既存レスを参照すると返信件数が増えるが、その時 JS は **既存の span の textContent を書き換えるだけ** にしている。span 自体が無いと位置が分からないので JS が末尾に append してしまい、テンプレで指定した位置とずれる。0 件のとき非表示にするのは `post.css` の `.post-reply-count:empty { display: none; }` で行う
- テンプレ中に文字列 `</script` を含めない (script タグ終端と誤認される)

**通常レス vs 埋め込みレス**: 単一テンプレ + CSS で出し分け。`{{#if isEmbedded}} embedded{{/if}}` で `.post.embedded` クラスを付加し、CSS 側で見た目を変える方針。`post.css` には `.post.embedded` 用のスタイル + `.inline-expansion.forward/reverse .post.embedded` の親/子別の色分けが入っている。

**実装位置**:
- `Services/Theme/ThemeService.cs` … 既定テンプレの展開と読込 (失敗時は埋め込みリソースから fallback)
- `Controls/WebView2Helper.cs::LoadThreadShellHtml` … スレシェル HTML に post.html を `<script type="text/template" id="post-template">` 内に、post.css を `<style>` 内に注入
- `Resources/thread.js` … `_parseTemplate` / `_renderTemplate` (~50 行) でテンプレを 1 度コンパイルし、`renderPost(data)` が `buildPostHtml` / `buildTreePostHtml` / `buildDedupPostHtml` から呼ばれる

#### 5.3.2 NG クイック追加ダイアログ (`NgQuickAddDialog`)

レス番号メニュー → 「NG → 名前 / ID / ワッチョイ」で開く軽量モーダル。本格的な編集は引き続き `NgWindow` (§7.2) で行うため、こちらは「ワンクリックで素早く 1 件追加」だけに特化。

| フィールド | 既定値 |
|---|---|
| 種別 (target) | クリックされた menu 子項目に対応 (`name` / `id` / `watchoi`)。`word` (本文) も選択肢として残す |
| パターン | JS 側で抽出済の値 (例: ID 文字列) をプリフィル。空でも可、ユーザが手で書ける |
| 方式 | `literal` (部分一致、既定) / `regex` |
| スコープ | この板のみ (既定) / 全板 (グローバル) |
| 期限 | **1 日 (既定)** / 1 週間 / 1 か月 / 無期限 |

OK で `NgRule` を生成し、呼出元 (`MainViewModel.OpenNgQuickAdd`) が `NgService.Save` で永続化する。

**実装上の注意 — WebView2 callback からの ShowDialog 再入回避**: `OpenNgQuickAdd` は WebView2 の `WebMessageReceived` 経路で呼ばれる。そこから直接 `ShowDialog` を叩くと WebView2 native 層の入力処理と modal 再入が競合し、稀に STATUS_BREAKPOINT (0x80000003) で落ちることがある。`Application.Current.Dispatcher.BeginInvoke` で 1 サイクル遅らせ、WebView2 の callback スタックを完全に巻き戻してから modal を始める。

### 5.4 リッチスクロールバー

マルチトラック方式。本文表示エリアの右側に複数の縦帯を並べる。

```
┌──┬──┬──┬──┬──┬──┐
│既│人│自│U │画│動│ ←各 3〜4px 幅
│読│気│分│R │像│画│
│境│レ│レ│L │  │  │
│界│ス│ス│  │  │  │
└──┴──┴──┴──┴──┴──┘
```

| トラック | 内容 | デフォ色 |
|---------|------|---------|
| 既読/未読境界 | 線 1 本(既読範囲の終端) | 灰 |
| 人気レス | 被アンカー数 ≥ 閾値(デフォ 3、設定可) | 赤 |
| 自分のレス | 自分の投稿 | 青 |
| URL | URL を含むレス | 緑 |
| 画像 | 画像を含むレス | 橙 |
| 動画 | 動画を含むレス | 紫 |
| 検索ヒット | Ctrl+F ヒット時に動的追加 | 黄 |

- 各帯の表示 ON/OFF/順序/色は設定可 (Phase 11 設定画面で実装予定、現状は固定)
- 実装はスレ表示 WebView2 内の HTML/CSS/JS (`#richScrollbar` div + `position: fixed`)。標準スクロールバー (右端 14px) の左に `right: 14px` で並べる
- マウスクリックでその位置にジャンプ (フラクション × scrollHeight)

**現在の実装状況**:

| トラック | 状態 | 補足 |
|---------|-----|-----|
| 既読/未読境界 (灰) | ✗ 撤去 | `scrollThumb` (現在表示範囲を半透明で示す) と機能が重複したため撤去 |

### 5.4.2 スレ一覧の列構成

`ThreadListHtmlBuilder` が組み立てる HTML テーブルの列:

| 列 | 内容 | ソート |
|---|---|---|
| 状態 (`col-log`) | 丸マーク (透明 / 青 = ログあり / 緑 = 新着 / **茶 = 落ちた**)。お気に入り済みスレでは背景に薄い ★ が重なる (`is-favorited`) | 数値 (`data-log`) |
| No | スレッド表示順 (subject.txt の行番号) | 数値 |
| タイトル | スレタイトル | 文字列 (`Intl.Collator('ja')`) |
| **板** | 板名 (`Board.BoardName`、お気に入り表示で板混在に対応) | 文字列 |
| 数 | レス数 | 数値 |
| 勢い | (1 日あたりレス数) | 数値 |

茶色マーク (`Dropped`) は subject.txt 不在のスレを強調する用途。出るシナリオ:
- お気に入りディレクトリ表示で、登録スレが subject.txt から消えた時
- **通常の板表示で、ローカルに dat があるが subject.txt にもう載っていないスレ** (= dat 落ちで保存だけ残っているスレ。タイトルは dat の 1 レス目から取得し、レス数は idx.json の lastFetchedPostCount から復元)

**列幅ドラッグ**: 各 `<th>` の右端に `<span class="col-resizer">` をホバーで現れる薄帯として配置し、ドラッグで列幅を変更できる。

- 永続化: `localStorage["chbrowser.threadlist.colwidths"]` に `data-sort` をキーとして保存 (例: `{ "log": 26, "title": 320 }`)
- 実装上の注意点 (落とし穴):
  - `<th>` は box-sizing: content-box (table cell の既定) + `padding: 4px 8px` のため、`getBoundingClientRect` の border-box 値をそのまま `style.width` に入れると padding 16px 分だけ境界がカーソルから飛ぶ → padding を引いた content-box 値で設定する
  - `table-layout: fixed; width: 100%` の table は 1 列の `style.width` を変えると table 全体を比例スケールする (= 操作していない列の境界も動く)。そのため**ドラッグ開始時に全列を現在のレンダー幅で inline 固定**したうえで、ドラッグ境界の左右 2 列だけを ±Δ で動かす方式にする。ドラッグ終了時は操作した 2 列以外の inline width をクリアして title 列の吸収モードに戻す
- ヘッダのクリックソートと衝突しないよう、resizer は `mousedown` で `stopPropagation` を呼ぶ
- `.col-log` (= 状態マーク列) のヘッダだけ縦スクロール時に追従しなくなる事象は `position: relative` がクラスセレクタとして指定されており `thead th { position: sticky }` を上書きしてしまうのが原因。`tbody td.col-log { position: relative }` に scope することで sticky を維持しつつ `.is-favorited::before` の絶対配置も担保している

### 5.4.3 左ペインの WebView 化 (Phase 9)

お気に入りペインと板一覧ペインを `<TreeView>` から WebView2 + 埋め込み HTML/CSS/JS にレンダリング移行済。スレ一覧で確立済みの「Html 添付プロパティ + JS への postMessage」パターンを踏襲。

**移行の動機**:
1. Edge ネイティブの Ctrl+F 検索が両ペインで使える (これがこのフェーズのメイン動機)
2. HTML/CSS を `Resources/` に外出ししてテーマ機構の対象に追加できる (Phase 11 / 12 の延長で活かす)
3. レンダラの統一によりコードベース全体の一貫性が増す

**設計判断**:
| 項目 | 採用案 | 理由 |
|---|---|---|
| 階層表示 | `<details>/<summary>` | Chromium ネイティブが Ctrl+F マッチ時に閉じた `<details>` を自動で `open=true` にする。`hidden="until-found"` も併用可 |
| コンテキストメニュー | WPF の `ContextMenu` をネイティブ表示 | デザインを変える必要が無い領域。HTML/CSS で再実装するより既存の Click ハンドラ / Command バインドを流用できて工数が縮む。JS の `contextmenu` を `preventDefault()` で抑制し、postMessage で C# に通知 → `PlacementMode=MousePoint` で開く |
| D&D | HTML5 Drag and Drop API | dragstart で source id を `dataTransfer` に詰めて、drop で `{type:'moveFavorite', sourceId, targetId, position}` を postMessage。循環防止は `FavoritesViewModel.CanReparent` で「移動後の新しい親」基準で検証 (= source 自身またはその子孫が新しい親になることを禁止) |
| 状態の正本 | ViewModel | 変更操作は JS → postMessage → MainViewModel 更新 → HTML 再生成 → push の単方向。スレ一覧と同じ単一チャネル設計 |
| 折りたたみ状態 | ViewModel に `IsExpanded` を持つ | HTML 再生成時に `<details open>` を出し分けて維持。ユーザがトグルしたら postMessage で同期 |

### 5.7 設定画面

#### 5.7.1 全体方針

- **即時反映**: ユーザが設定値を変更したら自動で `data/app/config.json` に保存し、その場で動作に反映する。OK / 適用 / キャンセルボタンは **持たない**
  - テキスト入力 (User-Agent など) は IME 確定 + フォーカス外し or 数百 ms の debounce で保存
  - 数値入力 (タイムアウト等) も同様
  - 即時反映できない項目 (HiDPI 等の再起動要) は次項参照
- **再起動が必要な変更のヘッダバナー**: HiDPI のような項目を変更すると、設定ウィンドウ上部に注意バナー「変更を反映するにはアプリの再起動が必要です [今すぐ再起動]」を表示。ボタンを押すと安全に終了 → 再起動。アプリ終了時にも未反映の設定があれば確認ダイアログ
- **検索ボックス**は将来検討 (項目数が増えてきた時点で導入)

#### 5.7.2 カテゴリと項目

| カテゴリ | 項目 |
|---|---|

#### 5.7.3 永続化

- ファイル: `data/app/config.json` (4.1 参照)
- スキーマは `Models/AppConfig.cs` (record) として実装。即時反映時に `ConfigStorage.Save` で atomic-tmp-rename で保存
- **後方互換**: アプリのバージョン更新で項目が増えたとき、未知のキーは無視 / 欠落キーは既定値補填する Json デシリアライズ。`Version` フィールドで将来のスキーマ移行用に番号管理 (1 から開始)

#### 5.7.4 デザイン編集の補足

- 「開く」ボタン押下時、対象 CSS がディスクに **未配置なら自動で既定 (= 埋め込みリソース) を書き出してから** 関連付けエディタで開く。ユーザは「開く」ボタンを押すだけで編集できる状態になる
- 「すべての CSS をリロード」を押した時の挙動:
  1. `WebView2Helper` 側の shell HTML キャッシュ (`_shellHtmlCache` / `_viewerShellHtmlCache`、HtmlBuilder 内の `_shellHtmlCache` 等) を全クリア
  2. 開いている全 WebView2 を再ナビゲート (各 ViewModel の `Html` / `ImageUrl` を一旦 null に → 再代入、もしくは付属の再描画 API 経由)
  3. スレ表示タブの場合は `idx.json.lastReadPostNumber` から自動でスクロール位置復元されるので、ユーザ体験的には「リロードしただけ」になる
- 「テーマフォルダを開く」 = `Process.Start("explorer.exe", themesDir)` で `data/themes/` を開く
- 「既定 CSS を生成」 = `ThemeService.ExtractDefaultThemeFiles` 相当を全 CSS に拡張したもの

### 5.8 ショートカット / マウスジェスチャー設定ウィンドウ (Phase 15)

`ツール → ショートカット & ジェスチャー...` で開く独立した別ウィンドウ (`Views/ShortcutsWindow`)。**モードレス + シングルトン + リサイズ可**、Owner = メインウィンドウ。設定ウィンドウとは別ウィンドウ (= NG 設定と同じ扱い、§5.7.2 表参照)。

#### 5.8.1 全体レイアウト

```

- 左カラム: **カテゴリ列** (`ShortcutAction.Category` の distinct を表示)。クリックで右側のリストをそのカテゴリに絞り込む。検索適用中はヒットを含むカテゴリのみハイライト
- 右カラム: アクションのリスト (4 列表示)
  - **アクション**: `DisplayName` (例: "現在のスレッドタブを閉じる")
  - **キー**: WPF Key 文字列 (`Ctrl+L` / `F5` / `Escape` など)。空欄クリックで入力受付モードに入り、次に押下されたキー組合せを記録
  - **マウス**: マウス操作のドロップダウン (24 オプション固定、§5.8.3 参照)
  - **ジェスチャー**: 右ドラッグで方向を入力するキャプチャボタン。ボタン押下中の右ドラッグで `←→↑↓` 列を記録 (`GestureRecognizer` と同じ量子化規則)
  - 行末にゴミ箱アイコン: その行の 3 列をすべてクリア (= "未割り当て" に戻す)
- 編集禁止セルの可視化: カテゴリと種別の組合せで設定不可なセル (= 「全体」のマウス入力 / 「スレ一覧のタブ領域」のジェスチャー、§5.8.3.1) は背景 `#FFFFFF` + 枠なし + カーソル `arrow` で表示 (= 設定可能セルと一目で区別)。ToolTip で理由を表示
- 上部
  - **検索ボックス**: 入力するたびにアクション名 / 現在の割当文字列のいずれかに含まれるかでリアルタイム絞り込み (差分計算ではなく ItemsSource の `CollectionViewSource.View.Refresh` ベース)
  - **既定に戻す**: ウィンドウ全体を `ShortcutRegistry` の `DefaultShortcut` / `DefaultMouse` / `DefaultGesture` で上書き (確認ダイアログ)
  - **保存**: 編集内容を `data/app/shortcuts.json` に書き出し、`ShortcutManager.Apply` で即時反映 (= ウィンドウを閉じなくても KeyBinding / 各 WebView へ push される)
- 終了時 (`Closing`) に未保存変更があれば確認ダイアログ ("保存して閉じる / 破棄して閉じる / キャンセル")

#### 5.8.2 1 アクション 1 行 1 設定の原則

各 `ShortcutAction` は 1 行で表現され、`(キー, マウス, ジェスチャー)` の 3 種を **同時に**最大 1 つずつ持てる。複数キーバインドを 1 アクションに紐付けたいケース (例: F5 でも Ctrl+R でも更新) は、現状は対応しない (=設定可能数が増えると UI が一気に複雑になるため、必要になった時点で「+」で追加できる行展開を再検討)。

##### 5.8.2.1 カテゴリ「全体」と編集禁止セル

`CategoryResolver.GlobalCategory = "全体"` は **画面のどこから入力しても発火**する仮想カテゴリ。`CategoryResolver.Resolve` はこの値を返さず、`ShortcutManager.Apply` の最後で「全体」のバインドを他全カテゴリ (`CategoryResolver.AllCategories`) のディスパッチマップにマージする (= ペイン側に同 descriptor が無いときだけ「全体」が効く)。

「全体」カテゴリ + 各カテゴリで以下の編集禁止ルールがある (`CategoryResolver.IsMouseEditable` / `IsGestureEditable` ヘルパで一元化):

| カテゴリ | マウス入力 | ジェスチャー |
|---|---|---|
| 全体 | **不可** (= 通常 UI 操作と衝突しやすい) | 可 |
| スレ一覧のタブ領域 | 可 | **不可** (= タブストリップ上の右ドラッグ判定が不安定) |
| その他 | 可 | 可 |

設定 UI 側は `ShortcutItem.CanEditMouse` / `CanEditGesture` プロパティを XAML の `IsEnabled` に bind してボタン無効化、ロード時には禁止セルの値を空に正規化して表示する。`ShortcutManager.Apply` でも同じ判定を再適用するので、`shortcuts.json` を直接編集して禁止組合せを書いても無視される (defense in depth)。

「全体」のキーボードショートカットは MainWindow + (attach 済なら) ViewerWindow の両方の `InputBindings` に登録され、どちらにフォーカスがあっても発火する。

##### 5.8.2.2 重複防止 (Phase 23)

各 `EditDialog` から OK が返ったタイミングで `FindBindingConflict(editing, newValue, kind)` をかけ、衝突を検出したら MessageBox で拒否する (= ダイアログ自体は閉じるが値は反映しない)。種別ごとの衝突ルール:

| 種別 | 衝突になる組合せ |
|---|---|
| ショートカット (キーボード) | 任意カテゴリ ↔ 任意カテゴリ (= アプリ全体で一意) |
| マウス入力 | 同カテゴリ内 / 「全体 ↔ ペイン」跨ぎ |
| マウスジェスチャー | 同カテゴリ内 / 「全体 ↔ ペイン」跨ぎ |

ペイン同士のマウス操作 / ジェスチャーは引き続き許容 (= スコープが分離されているので衝突しない)。実装は `Views/ShortcutsWindow.xaml.cs` の `BindingKind` enum + 共通ヘルパに集約。

#### 5.8.3 マウス操作のドロップダウン (24 オプション固定)

ユーザがフリーフォームでマウス操作を入力できると入力ミス + 検証コストが大きいため、ドロップダウン固定にしている。詳細は `Services/Shortcuts/MouseOperationCatalog.cs`。

| グループ | オプション |
|---|---|
| (未設定) | 空文字 |
| 単体 | 中クリック / ホイールアップ / ホイールダウン / ダブルクリック / トリプルクリック |
| Shift + | 左クリック / 右クリック / 中クリック / ホイールアップ / ホイールダウン |
| Alt + | 左クリック / 右クリック / 中クリック / ホイールアップ / ホイールダウン |
| Ctrl + | 左クリック / 右クリック / 中クリック / ホイールアップ / ホイールダウン |
| 右クリック chord | 中ボタン / ホイールアップ / ホイールダウン |

- 修飾なしの「左クリック」「右クリック」は通常 UI 操作 (アクティブ化 / コンテキストメニュー) と衝突するため**意図的に外してある**
- 「右クリック + ...」はマウスチョード (= 右ボタン押下中に副操作)。`ShortcutManager` 側で `Mouse.RightButton == Pressed` を主とし、JS bridge 側は `event.buttons & 2` (MK_RBUTTON) で判定。WPF / WebView2 境界をまたいで右ボタンを保持し続けても認識される (§7.8 参照)

#### 5.8.4 永続化

- ファイル: `data/app/shortcuts.json` (UTF-8、atomic-tmp-rename)
- スキーマ:
  ```json
  {
    "version": 1,
    "bindings": {
      "main.focus_address_bar":      { "key": "Ctrl+L",   "mouse": "",                  "gesture": "" },
      "thread.close_current_in_body":{ "key": "Ctrl+W",   "mouse": "中クリック",         "gesture": "↓→" },
      "viewer.next_image":           { "key": "Right",    "mouse": "ホイールダウン",     "gesture": "→" }
    }
  }
  ```
- 読込時: `bindings` に未掲載のアクションは `ShortcutRegistry` の既定値を使う。掲載済アクションは `key/mouse/gesture` のいずれかが空なら「ユーザがクリアしたものとして扱う」(= 既定値で埋め直さない)
- アプリのバージョン更新で新アクションが増えた場合: 既存 `shortcuts.json` を読み込んだ後に新アクションは既定値で補填される (UI に表示するときも既定がそのまま見える)
- 衝突検出は §5.8.2.2 で実装済 (= 編集ダイアログ OK 時に拒否)。`shortcuts.json` を直接編集して衝突状態にした場合の検出は将来検討 (= 現状は `ShortcutManager.Apply` で descriptor が後勝ち上書きされる)

## 6. アーキテクチャ

### 6.1 全体方針

- **MVVM + WebView2 ハイブリッド**
- WPF レイヤー: ウィンドウ / メニュー / ツールバー / アドレスバー / タブ管理 / ステータスバー。ViewModel が `INotifyPropertyChanged` で状態を露出
- WebView2 レイヤー: 板一覧 / お気に入り / スレ一覧 / スレ表示 / ビューアの全コンテンツ描画。`Resources/` の埋め込み HTML/CSS/JS が DOM を構築 (Phase 9 で左ペインも WebView 化済)
- C# ↔ JS のブリッジは添付プロパティ + `CoreWebView2.PostWebMessageAsJson` (C→J) と `window.chrome.webview.postMessage` (J→C)
- ショートカット / マウス操作 / マウスジェスチャーは `ShortcutManager` 中央集権 + `shortcut-bridge.js` で WPF / WebView の両側からディスパッチ (Phase 16、§7.9 参照)
- `Service` 層が永続化・通信を担当
- DI: 軽量手書き (`App.OnStartup` でコンストラクタ注入)
- 非同期: `async/await` 全面採用

### 6.2.1 スレ表示シェルのナビゲーション (履歴メモ)

スレ表示シェル HTML は `CoreWebView2.NavigateToString` で配信している。

**YouTube Error 153 への対応試行履歴**:
- 当初 `NavigateToString` で生成されるページの origin が `null` (data: URL) になり YouTube embed が `Error 153` を出していたため、以下を順に試した:
  1. `WebResourceRequested` で `https://chbrowser.local/thread.html` の Document コンテキストを捕捉して埋め込み HTML を返す方式 → 現行 SDK では top-level navigation を一貫して捕捉できず、ページが空白になる
  2. `SetVirtualHostNameToFolderMapping` で同じ仮想ホストを temp フォルダに紐付け、`Navigate(ShellUri)` で読み込む方式 → 一瞬表示後に画面が真っ白になる事象が発生 (原因未特定)
- いずれも安定動作せず、`NavigateToString` に revert した
- YouTube embed を `youtube-nocookie.com/embed/<id>?autoplay=1&playsinline=1` (プライバシー強化版) に変更したのは維持。これだけで多くの動画は再生できる
- Error 153 は per-video の embed 制限 (uploader 設定) でも出るため、すべての動画で出るのか特定の動画だけなのかは要確認

## 7. 主要アルゴリズム

### 7.1 アンカー解析

JS 側 (`thread.js`) の正規表現で本文中のアンカーを認識する:

```
>>\s*(\d+)(?:\s*-\s*(\d+))?      ; 半角 + 範囲 (>>1-5)
read.cgi/<board>/<key>/(\d+)(?:-(\d+))?/?$  ; 5ch dat 由来 <a href> から N を抽出
```

- `>>N` 単発 / `>>N-M` 範囲のみ対応 (現状)
- `>>N,M` のカンマ列挙、全角 `＞＞` は未対応 (Phase 拡張)
- 5ch dat に埋め込まれる `<a href="../test/read.cgi/board/key/N">>>N</a>` は inner text と href のどちらからでも抽出可
- 表示時にリンク化 (`<a class="anchor" data-from data-to>`)、ホバーで多段ポップアップ、クリックで scrollIntoView

### 7.2 NG 機能 + 連鎖あぼーん (Phase 13)

**データモデル** (`Models/NgRule.cs`):
```
NgRule {
    Id: Guid (= ルール ID)
    Target: "name" | "id" | "watchoi" | "word"     // どのフィールドに対する NG か
    MatchKind: "literal" | "regex"                 // 通常文字列 (部分一致) / 正規表現
    Pattern: string                                 // 検索文字列または正規表現
    Enabled: bool                                   // 個別の有効/無効トグル
    CreatedAt: DateTimeOffset
    ExpiresAt: DateTimeOffset?                      // null = 無期限、それ以外は YYYY-MM-DD 終端
    Memo: string                                    // 任意のメモ (オプション)
}
```

**永続化** (Phase 13e で単一ファイル化):
- 全ルール: `data/ng/rules.json` の `Rules` 配列。各ルールは `Scope` フィールドで `"global"` または板単位 (`{ rootDomain, directoryName }`) を表現
- 旧形式 (`data/ng/global.json` + `data/ng/by_board/<root>__<dir>.json`) からは `NgStorage.LoadAndMigrate` が起動時に自動移行 + 旧ファイル削除

**フィールド抽出** (NgService 内):
- `Target="name"` → `Post.Name` をそのまま subject に
- `Target="id"` → `Post.IdField` (=日付フィールドの末尾 `ID:xxx` 部分。dat 仕様参照) から ID 部分を抽出
- `Target="watchoi"` → ID フィールド内から `[A-Za-z0-9]{4}-[A-Za-z0-9]{4}` (= ハイフン区切りの 8 桁英数 hex) を抽出。bone 名 (ワッチョイ / オッペケ / アウアウウー / 等) は無視 — 8 桁本体だけで判定
- `Target="word"` → `Post.Body` (本文 HTML/プレーン) を subject に

**判定**:
1. 適用ルール集合 = グローバル ∪ そのスレが属する板の単位ルール
2. 期限切れ (ExpiresAt < 現在時刻) を除外
3. Enabled = false を除外
4. 各レスについて、ルールごとに subject を抽出 → MatchKind に応じて部分一致 / Regex マッチ
5. どれか 1 つマッチしたらヒット

**連鎖あぼーん** (無限再帰、fixpoint まで):
```
hidden = NG ルールに直接マッチするレス番号集合
loop:
  changed = false
  for post in 全レス:
    if post.number in hidden: continue
    if post 本文中のアンカー (>>N) のうち hidden に含まれるものが 1 件以上:
      hidden.add(post.number); changed = true
until not changed
```

**正規表現バリデーション**:
- 追加 / 編集時に `Regex.Match("", pattern)` で構文チェック
- 失敗時 → ダイアログ「正規表現が不正なため、この NG ルールは保存しますが有効化できません: <例外メッセージ>」 + `Enabled=false` で保存
- 後にユーザがトグルで有効化しようとしたとき、再度バリデーション → 失敗ならダイアログ + トグルは無効のまま

**スコープ**:
- グローバルルールは全スレで適用
- 板単位ルールは (Board.Host のルートドメイン, Board.DirectoryName) が一致するスレで適用

**スレ表示への組み込み** (透明あぼーん固定):
- `MainViewModel.OpenThreadAsync` 内で dat パース後、Posts 配列に対して NgService で hidden 集合を計算
- hidden な Post は **JS への送信 (`appendPosts`) から除外** — JS 側は hidden の存在を知らないし、DOM にも入らない
- 結果として連番は飛ぶが、レス番号の数値は dat 由来なので一貫性は保たれる

**件数表示**:
- ステータスバーに「あぼーん N」を表示 (= 現在開いているスレタブの hidden 件数)
- タブ切替 / NG 設定変更時に更新

**NG 設定 UI** (Phase 13b):
- **設定ウィンドウから NG カテゴリは外し、独立した別ウィンドウ** (`NgWindow`) として実装
- `ツール → NG 設定...` で開く。**モードレス + シングルトン + リサイズ可**
- グローバル / 板単位タブ、ルール一覧 (DataGrid)、追加 / 編集 / 削除、有効化トグル

### 7.3 被アンカー集計(人気レス)

JS の `buildReverseIndex()` がレンダリング時に逆引き Map を構築する。各レスについて本文中の `>>N` を抽出し、`Map<被参照レス番号, 参照元レス番号配列>` を作る。リッチスクロールバーの人気レストラックは `Map[N].length >= 3` のレスを赤マーカーで表示。

将来的には `idx.json.anchorCount` にキャッシュして再計算コストを削減予定 (現状は表示のたびに O(N) で再計算; 数千レスでも十分高速)。

### 7.4 画像 / 動画 URL 抽出

レス本文の URL を JS が走査し、以下の優先度で判定する:

1. **YouTube** (`youtube.com/watch?v=`、`youtu.be/`、`youtube.com/shorts/`、`youtube.com/embed/`) — `extractYouTubeId(url)` が 11 桁 ID を返す。サムネ `i.ytimg.com/vi/<id>/hqdefault.jpg` を `image-slot.youtube` で表示。クリックは `postOpenUrl` でデフォルトブラウザに飛ばす (インライン iframe 再生は YouTube embed の Error 153 が安定して出るため不採用、Phase 10 のビューアウィンドウで本格対応を検討予定)
2. **直リンク動画** (`.mp4 .webm .mov`) — `VIDEO_EXT_RE` 一致。`image-slot.video` で 320×180 placeholder + 再生アイコン。viewport 近接で `materializeVideoSlot` が `<video preload="metadata" muted>` を埋め込み、`loadedmetadata` で 0.1s シークして first frame をサムネとして表示。クリックで `controls + unmute + play()` (`playMedia`)。CORS や seek 失敗時はサムネが出ないが、再生フローは継続
3. **画像直リンク** (`.jpg .jpeg .png .gif .webp`) — `IMAGE_EXT_RE` 一致。`image-slot` で 240×240 のサムネ、HEAD 後に `<img>` で表示
4. **同期展開対象** (imgur 等) — `URL_EXPANDERS` で URL 形だけで実体画像 URL を決定
5. **非同期展開対象** (x.com / pixiv) — `ASYNC_EXPANDER_RES` に一致 → `image-slot.async` を出して、`imageMetaRequest` の応答 (`resolvedUrl`) で `data-src` を上書きしてから HEAD/cache → `<img>` 化

ページ URL → 実体画像 URL の展開は JS の `URL_EXPANDERS` 配列に変換器を並べる方式:

| URL パターン | 展開結果 | 状態 |
|---|---|---|
| `imgur.com/<id>` (album/gallery/tag を除く) | `i.imgur.com/<id>.jpg` | ✅ JS の `URL_EXPANDERS` で同期展開 (URL 形だけで決まる) |
| `(twitter\|x\|fxtwitter\|vxtwitter).com/<user>/status/<id>` | `pbs.twimg.com/media/...` (1 枚目) | ✅ C# `UrlExpander` が `api.fxtwitter.com/i/status/<id>` を JSON 取得して 1 枚目を返す |
| `pixiv.net/(en/)?artworks/<id>` | `i.pximg.net/img-master/.../regular/...` | ✅ C# `UrlExpander` が `www.pixiv.net/ajax/illust/<id>` を JSON 取得して `urls.regular` を返す。R-18 / 非公開はログイン要のため失敗することあり |
| `youtube.com/watch?v=<id>` / `youtu.be/<id>` / `youtube.com/shorts/<id>` / `youtube.com/embed/<id>` | サムネ `i.ytimg.com/vi/<id>/hqdefault.jpg` 表示のみ。クリックでデフォルトブラウザに開く | ✅ JS で URL から ID を抽出してサムネ表示。インライン iframe 再生は YouTube embed の Error 153 が安定して出るため採用しない (Phase 10 のビューアウィンドウで対応するか検討中) |

JS 側は同期展開と非同期展開を区別:
- **同期** (`getInlineImageSrc(url)`): 直リンク or imgur 単純展開 → 即 `data-src` に画像 URL を入れる
- **非同期** (`isExpandableAsync(url)`): x.com / pixiv → `data-async-expand="1"` 付きで slot を出し、`imageMetaRequest` 応答の `resolvedUrl` で `data-src` を上書きしてからロード

リッチスクロールバーの画像トラック (橙) は `bodyContainsImage` が本文中の URL 全てに対して `getInlineImageSrc` または `isExpandableAsync` を試すので、imgur / x.com / pixiv URL を含むレスは橙マーカーが立つ。

C# 側 `Services/Image/UrlExpander.cs` のポリシー:
- 専用 `HttpClient` (User-Agent: ブラウザ風、Accept-Language: `ja,en`)
- `IsAsyncExpandable(url)` は静的判定、`ExpandAsync(url)` は in-memory `ConcurrentDictionary<url, Task<string?>>` でセッションキャッシュ (同じ URL の同時呼び出しは Task を共有)
- 失敗 (媒体無し / API エラー / login 要) は null を返す → JS 側で `expand-failed` プレースホルダ

pixiv の i.pximg.net は CDN 側で `Referer: https://www.pixiv.net/` を要求するため、`WebView2Helper.InstallImageCacheHandlers` が `WebResourceRequested` 内で `*.pximg.net` 行きの URL に `Referer` ヘッダを差し込む (HtmlAgilityPack 等は不要、API 応答 + ヘッダ修正で完結)。

### 7.5 画像サイズしきい値 (5MB) と遅延ロード

帯域節約のため、画像 URL が見つかってもすぐには `<img src>` を出さない。代わりに 240×240 のスロット (`<span class="image-slot deferred">`) を出しておき、以下のフローで実体ロードを判断する:

1. JS の `IntersectionObserver` (rootMargin: 300px) がスロットを viewport 近接で検出
2. JS が `imageMetaRequest` メッセージで C# に HEAD/cache 確認を要求
3. C# 側の判定:
   - **ローカルキャッシュにあり** → HEAD スキップして `{ ok: true, cached: true }` を返す
   - **無し** → `Services/Image/ImageMetaService` が HEAD で `Content-Length` を取得 (失敗時は `ok: false, size: null`)
4. C# が `imageMeta` メッセージで結果を JS に返す
5. JS の `applyImageMeta`:
   - **cached: true** → しきい値無視で即ロード (帯域コスト 0)
   - size ≤ 5MB → `<img src=...>` を生成して通常ロード
   - size > 5MB → `image-slot` に `over-threshold` クラスと「X.X MB - クリックで読み込む」テキスト
   - HEAD 失敗 (size 不明) → そのまま読み込む (HEAD 非対応サーバーで全画像が出ないのを防ぐ)
6. ユーザーが `over-threshold` スロットをクリック → JS `loadSlotImage` で強制ロード

`ImageMetaService` は in-memory `ConcurrentDictionary<string, Task<ImageMeta>>` でキャッシュし、同じ URL は HEAD 1 回のみ。同時 HEAD は `SemaphoreSlim` で 6 本までに制限。User-Agent はブラウザ風 (`Mozilla/5.0 ... ChBrowser/0.1`)。

しきい値 5MB は現状 JS 側に固定値 (`IMAGE_SIZE_THRESHOLD`)。Phase 11 設定画面で調整可能化予定 (config.json に保存し、起動時に setConfig メッセージで JS に push する想定)。

### 7.6 画像ローカルキャッシュ (透過)

`Services/Image/ImageCacheService` がディスクキャッシュを担当し、`WebView2Helper` が WebView2 のリソースリクエストに介在することで、JS 側からは何も意識せずキャッシュが効く構造にしている。

- `WebView2Helper.RegisterImageCache(cache)` を App.OnStartup で 1 度呼ぶ
- 各 WebView2 が `EnsureCoreAsync` で初期化されるタイミングで `InstallImageCacheHandlers` がハンドラを取り付ける (`ConditionalWeakTable` で 1 度だけ)
- `CoreWebView2.AddWebResourceRequestedFilter("*", Image)` で画像コンテキストの全リクエストを介在対象に
- `WebResourceRequested`: キャッシュヒットなら `File.ReadAllBytes` → `MemoryStream` で `CreateWebResourceResponse(stream, 200, "OK", "Content-Type: ...")` をセット (ネットワークに出さない)
- `WebResourceResponseReceived`: ステータス 200 + Method GET + `Content-Type: image/*` の応答を `GetContentAsync()` でストリーム読み出し → `ImageCacheService.SaveAsync` でディスクに保存

JS 側の `<img>` 生成は素直な `<img src="https://i.imgur.com/abc.jpg">` のまま。WebView2 が透過介在するので JS 側にキャッシュロジックは不要。

`<img src>` を出すかどうかの判定は `imageMeta` の `cached` フラグでも併用する。HEAD を打たずキャッシュ参照だけで済めば往復遅延も省ける。

### 7.7 既読判定 (実装済み: ビューポート上端ベース)

- JS の `findTopmostVisiblePostNumber()` がビューポート上端のレス番号を返す
- スクロール時に 200ms debounce で上端レス番号を C# に postMessage (`scrollPosition`)
- C# `MainViewModel.UpdateScrollPosition` が `idx.json.lastReadPostNumber` を更新 + メモリ上の `tab.ScrollTargetPostNumber` も更新
- 次回オープン時、`OpenThreadAsync` が idx.json を読んで `tab.ScrollTargetPostNumber` を初期化、`appendPosts(scrollTarget)` メッセージに同梱して JS に渡す。streaming 中に対象レスが DOM に出現した時点で `scrollIntoView` で復元

設計書旧案 (`read_until` = 既読範囲の終端) との差異: 現状は「現在の上端レス」を保存している。一方「これまでに見た最大のレス番号」(= 最深既読位置) は Phase 19 で別フィールド `lastReadMarkPostNumber` として独立に追跡するようになった (詳細は 7.13 「ここまで読んだ」帯)。

### 7.8 URL 解析 (アドレスバー、Phase 14)

`Services/Url/AddressBarParser.cs` がアドレスバーへの入力テキストを以下に分類する。

```
AddressBarTarget {
    Kind: Board | Thread | Invalid
    Host: string         // 完全 host (例: "news.5ch.io") — Invalid 時は空
    Directory: string    // 板 dir (例: "news")           — Invalid 時は空
    ThreadKey: string    // dat 番号                       — Kind=Thread のときのみ
}
```

**解析手順**:
1. 入力 trim、空なら Invalid
2. プロトコル省略 (例: `news.5ch.io/news/`) は `https://` を前置
3. host が `*.5ch.net` なら `*.5ch.io` に書き換え (それ以外のホスト書き換えは行わない)
4. `Uri.TryCreate` で URL として parse 失敗なら Invalid
5. host が `*.5ch.io` または `*.bbspink.com` でなければ Invalid
6. パス解析 (末尾スラッシュ optional):
   - `^/<dir>/?$`                              → Kind=Board
   - `^/test/read.cgi/<dir>/<key>(/.*)?$`      → Kind=Thread (suffix `/100-200` 等は無視 — 将来「指定レスへスクロール」機能の入口として再利用する余地)
   - 上記いずれにも合わない                    → Invalid
7. `<dir>` は `[A-Za-z0-9]+`、`<key>` は `[0-9]+` の制約 (それ以外は Invalid)

**ナビゲーション** (`MainViewModel`):
- Kind=Board → `OpenBoardByUrlAsync(host, dir)`
  - 既存 ThreadListTab を `(ExtractRootDomain(host), dir)` で検索 → 一致すれば `SelectedThreadListTab = それ`
  - 無ければ `ResolveBoard(host, dir, "")` で Board を組み立てて既存の `LoadThreadListAsync` 経由で開く
- Kind=Thread → `OpenThreadByUrlAsync(host, dir, key)`
  - 既存 ThreadTab を `(ExtractRootDomain(host), dir, key)` で検索 → 一致すれば `SelectedThreadTab = それ`
  - 無ければ既存の `OpenThreadFromListAsync(host, dir, key, "", null)` で開く
- Kind=Invalid → アドレスバーを赤枠化 + ステータスバーへエラー文 ("認識できない URL です" 等)、入力テキストは保持

**root domain 比較の根拠**: 5ch.io / bbspink.com の各板は (root domain, dir) で一意 (= 同じ dir が複数のサブドメインに重複登録されない)。サブドメイン込みで比較するメリットは無く、ユーザがサブドメイン違いの URL を貼ったときにタブにヒットしない罠を避けるため、root 比較に倒す。Phase 13 の NG スコープ判定と同じ `DataPaths.ExtractRootDomain` を流用する。

**実装ファイル**:
- 新規 `Services/Url/AddressBarParser.cs` (純粋関数、static の `TryParse`)
- `MainWindow.xaml` メニュー行のレイアウト改修、`TextBox` 追加、`Ctrl+L` の `KeyBinding`
- `MainWindow.xaml.cs` 「貼り付けて移動」、ペイン focus 検出 → アドレスバーリセット
- `ViewModels/MainViewModel.cs` `AddressBarUrl` / `AddressBarHasError` / `OpenBoardByUrlAsync` / `OpenThreadByUrlAsync` / last-activated 追跡

### 7.9 ショートカット / マウスジェスチャーのディスパッチ (Phase 16)

設定値 (`shortcuts.json` + `ShortcutRegistry`) → 各ウィンドウ / 各 WebView へバインド配信 → 入力捕捉 → アクション実行までの一連のフロー。実装は `Services/Shortcuts/`。

#### 7.8.1 中央集権 `ShortcutManager`

- インスタンス: アプリ全体で 1 つ。`App.xaml.cs` で生成し、メインウィンドウ / ビューアウィンドウを attach
- 入力データ:
  - `(category, descriptor) → actionId` の二段マップ (`_inputsByCategory`)。category は `ShortcutAction.Category` (= "全体" / "メインウィンドウ" / "お気に入りペイン" / "板一覧ペイン" / "スレ一覧表示領域" / "スレ一覧のタブ領域" / "スレッド表示領域" / "スレッドタブ表示領域" / "ビューアウィンドウ")。descriptor はキー / マウス / ジェスチャーの**統一文字列**
  - `actionId → handler` の単一マップ (`_handlers`)。`App.BuildShortcutHandlers` で構築。VM の Command 呼び出しが多いが、JS 側で完結する処理 (viewer のズーム / 回転) は no-op を入れて「JS の localActions に逃がす」サインにする
- `Apply(ShortcutSettings)`:
  1. 全ウィンドウから既存 `KeyBinding` を撤去
  2. 全 `ShortcutAction` を回り、`(shortcut, mouse, gesture)` の 3 種を `_inputsByCategory[category]` に追加 (キーボードもここに入れる — JS 側で同じマップを再利用するため)。`CategoryResolver.IsMouseEditable` / `IsGestureEditable` で false のものはここで弾く (defense in depth)
  3. キーバインドを category に応じたウィンドウへ登録: `_mainWindow` (通常) / `_viewerWindow` (= "ビューアウィンドウ") / **両方** (= "全体" — どちらにフォーカスがあっても発火する)
  4. **「全体」マージ**: `_inputsByCategory["全体"]` が存在すれば、`CategoryResolver.AllCategories` の他全カテゴリの inner dict に「まだ存在しないキーだけ」追加 (= ペイン側の同入力バインドが勝つ semantics)。これにより 1 つ「全体」に bind するだけで全ペインから発火する
  5. `OnBindingsApplied` callback で全 category の descriptor → actionId スナップショットを通知。各 WebView (= JS bridge) はマージ済マップを受けて `setShortcutBindings` で自分のカテゴリ分を再 push
- ディスパッチ: WPF / WebView 由来どちらも `Dispatch(category, descriptor)` または `DispatchGesture(startCategory, gesture, source)` の二経路を通る。両者とも `_inputsByCategory` lookup 後、`_handlers[actionId]` を invoke

#### 7.8.2 descriptor の統一フォーマット

descriptor 文字列は形が互いに重ならないので、キー / マウス / ジェスチャーを 1 つの dictionary に混ぜても衝突しない:

| 種別 | 例 | 構築場所 |
|---|---|---|
| キー | `Ctrl+L`、`F5`、`Escape`、`Right` | WPF: `ShortcutManager.BuildModifiedKey` / JS: `shortcut-bridge.js` の `keyNameMap` (event.code → WPF Key 名) |
| マウス | `中クリック`、`Ctrl+ホイールアップ`、`Shift+左クリック`、`ダブルクリック` | WPF: `BuildModifiedKey` / JS: bridge 内で同名生成 |
| マウスチョード | `右クリック+中ボタン`、`右クリック+ホイールアップ` | 右ボタン押下中に他ボタン/ホイールが発火した時に組み立て |
| ジェスチャー | `↓→`、`←↑`、`→` | `GestureRecognizer` (WPF) / bridge の `directions` 配列 (JS)、いずれも 18px 量子化 + 連続同方向マージ |

JS 側 (bridge) と C# 側 (`ShortcutManager`) は**同じ綴り** (Ctrl→Alt→Shift→Win の順、`+` 区切り、ボタン名は日本語) を使う。これにより 1 つの `(category, descriptor) → actionId` マップを両側で共用でき、新しい入力種別を追加するときも片側だけ書けば済む。

#### 7.8.3 カテゴリ判定 (= スコープ解決)

入力イベントの**発生位置**から category を割り出すのが基本方針。「全体」カテゴリのバインドは `Apply` で全カテゴリへマージされるため、resolve 結果がペインカテゴリでも該当 descriptor が見つかれば「全体」のアクションが発火する (= 真のグローバル経路はこれで実現)。

- WPF イベント (`PreviewMouseDown` / `PreviewMouseWheel`): `e.OriginalSource` を `CategoryResolver.Resolve` に渡してビジュアル/論理ツリーを上向きに辿り、最初に見つかった category 名を返す (例: スレタブ TabPanel → "スレッドタブ表示領域")。どこにも当たらないとフォールバックで "メインウィンドウ" を返す (= chrome / メニュー / アドレスバー上)
  - `CategoryResolver.Resolve` は **"全体" を返さない**。「全体」は dispatch 時のマージ結果としてのみ表れる仮想カテゴリ
  - `Run` (TextElement) など `Visual` でない要素は `VisualTreeHelper.GetParent` が例外を出すため、`LogicalTreeHelper.GetParent` にフォールバックする `GetAnyParent` ヘルパで吸収
- WebView 内入力 (= JS bridge): WebView 自体が DataContext に `ThreadTabViewModel` / `ThreadListTabViewModel` / `ImageViewerTabViewModel` 等を持っているので、それで category が決まる。bridgeReady 受信時に `mgr.GetBindingsForCategory(category)` でマージ済マップ subset を push する (= 全体バインドも自動で含まれる)
- ジェスチャー: **開始時点**の category を `GestureRecognizer` (WPF) / bridge (JS) が確定し、終了まで保持する。途中でカーソルが別ペインに入っても category は変わらない (= 「画像ビューア起動 → スレ表示の上を通過 → ジェスチャー終了」しても画像ビューアのバインドが起動する)

#### 7.8.4 タブターゲット (mouse 由来は SelectedTab フォールバックしない)

タブストリップ上で「タブ B を中クリック」した時、操作対象は **クリックされたタブ B** であって `SelectedTab` (= タブ A) ではない。

- `App.BuildShortcutHandlers` の close/refresh などは `ResolveTargetThreadTab(source, vm)` ヘルパで決定:
  - `source != null` (= マウス由来) → ビジュアルツリーを辿って `TabItem.DataContext` の `ThreadTabViewModel` を返す。見つからなければ **null** (= 何もしない)
  - `source == null` (= キー由来) → `vm.SelectedThreadTab` にフォールバック
- これによりキーボードの「現在のタブを閉じる」は意図通り SelectedTab を閉じ、マウスの「中クリックでタブを閉じる」はクリックしたタブを閉じる、という両立が成立する

#### 7.8.5 WebView 内入力のキャプチャ (`shortcut-bridge.js`)

WebView2 (HwndHost) 内の入力イベントは Win32 サーフェスに吸われ WPF の routed event には届かない。各ペインの主 JS が `window.createShortcutBridge({...})` を呼んで bridge を初期化し、以下を担当:

- `document.addEventListener(..., capture=true)` で keydown / mousedown / mousemove / mouseup / contextmenu / wheel をキャプチャフェーズで横取り
- 受信した `setShortcutBindings` (C# → JS) で descriptor → actionId マップを保持。bind 済 descriptor の入力のみ `preventDefault + stopPropagation` して C# に shortcut/gesture メッセージを送る
- `localActions[actionId]` が定義されていれば C# 往復なしで JS 側で実行 (例: viewer の zoom_in/out/rotate_*、thread のスクロール)
- ジェスチャー: 18px 量子化 + 連続同方向マージで `↑↓←→` を生成。途中経過は `gestureProgress` メッセージでステータスバー表示用に流す
- 右クリックチョード: bridge 内の `rightDragging` フラグに加えて `event.buttons & 2` (MK_RBUTTON) もチェック。WPF→WebView や WebView→WPF を跨いで右ボタンを保持し続けても認識可
- 通常の右クリック (移動なし) は contextmenu を抑止しない。ジェスチャー認識時のみ `suppressNextContext = true` で次の contextmenu を抑止

##### 7.8.5.1 bindings の単方向 push (`bridgeReady`)

WebView は再ナビゲーションやタブ再生成で初期化される。シンプルに「`PaneShortcutsJson` 添付プロパティで全 WebView に push」する設計だと、

- 添付プロパティは値が**変化したとき**しか発火しないので、設定変更がなければ新規 WebView にバインドが届かない
- 各ペイン JS の中で `setShortcutBindings` を待つ実装にしてしまうと、`if (!tbody) return` のような早期リターンの後ろに置いた listener には届かない

の 2 つの落とし穴がある。これを回避するため、

1. `setShortcutBindings` を受信する listener は**ペイン JS ではなく `shortcut-bridge.js` 内に**置く (= bridge を呼んだ瞬間に listener が必ず動いている)
2. bridge が初期化完了で `{ type: 'bridgeReady' }` を C# に postMessage
3. C# 側 (各 `*WebMessageReceived`) は `bridgeReady` を受けたら `mgr.GetBindingsForCategory(category)` で当該カテゴリの subset を `PostWebMessageAsJson` で **その WebView だけ** に直接 push

これにより「PaneShortcutsJson が値変化していない」「タブ再生成で property change が発火しない」ケースでも binding が確実に届く。

#### 7.8.6 ステータスバーへのジェスチャー進捗表示

ジェスチャー入力中はステータスバー右側に `[<カテゴリ>] <方向列>` を表示する (例: `[スレッド表示領域] ↓→`)。

- `ShortcutManager.NotifyGestureProgress(category, gesture)` を WPF (`GestureRecognizer`) / 各 WebView (`gestureProgress` メッセージ受信) の両方から呼ぶ
- マネージャは `OnGestureProgress` callback で `MainViewModel.UpdateGestureStatus` に流す
- `gestureEnd` (`category=null`) でクリア

#### 7.8.7 複数ウィンドウ対応

メインウィンドウとビューアウィンドウが独立した `Window.InputBindings` を持つ。`ShortcutManager._appliedKeyBindingsByWindow: Dictionary<Window, List<KeyBinding>>` で各ウィンドウごとに登録済 KeyBinding を追跡し、`Apply` 時に全ウィンドウから一斉撤去 → カテゴリに応じたウィンドウへ再登録、という流れ。

ビューアウィンドウは `App.ShowImageInViewer` で初回表示直前に `_shortcutManager.AttachViewerWindow(_imageViewerWindow)` を呼んで attach する。attach 時に `_lastSettings` がキャッシュされていれば `Apply` を再実行して "ビューアウィンドウ" カテゴリの KeyBinding を viewer に流し込む。

ビューアウィンドウのマウス系入力 (ホイールでタブ切替 / Ctrl+ホイールでズーム / 90 度回転) は viewer 自身の WebView 内で発生するため、bridge → WebMessageReceived → `mgr.Dispatch("ビューアウィンドウ", descriptor)` 経由でディスパッチされる。viewer ウィンドウの chrome (WebView 外) でのマウス操作は現状サポートしない。

### 7.10 お気に入りチェック (Phase 17)

お気に入り登録スレに新着があるかを一括確認し、新着があるスレを差分取得しつつ新規タブで開く処理。`MainViewModel.CheckFavoritesAsync` が実体。

**入口** (3 箇所、すべて同じ `MainWindow.FavCheckUpdates_Click` を経由):
- お気に入りペイン上部のリフレッシュボタン
- 仮想ルートの右クリックメニュー → 「お気に入りチェック」
- メニューバーの ファイル → 「お気に入りチェック」

**フェーズ構成**:

1. **エントリ収集**: `CollectFavoriteEntriesFromVm` でルートから再帰的に board/thread をフラット化。スレ 0 件ならスキップ
2. **出元板の解決**: 登録スレの (host, dir) のユニーク集合を作って `Board` を解決。fav board エントリは無視 (= 登録板の中身は引かない、Phase 17 仕様)
3. **subject.txt 並列取得**: 共有 `SemaphoreSlim(BatchConcurrency)` 配下で `_subjectClient.FetchAndSaveAsync` を `Task.WhenAll`。`Interlocked.Increment` でカウンタを増やし `Dispatcher.InvokeAsync` で StatusMessage に「subject 取得 N/M」を反映
4. **新着判定**: 各登録スレについて `(subject.txt 内 PostCount > idx.json.LastFetchedPostCount)` なら `toOpen` に追加。subject.txt に無い (= 落ちた) スレは **除外** (Phase 17 仕様、Phase 7 の「落ちたら開く」を撤回)
5. **dat 並列取得**: `toOpen` 全件の `_datClient.FetchStreamingAsync` を **同じ semaphore** 配下で並列起動。Task list を `foreach + await` で開始順に処理 (= 早く完了したタスクは即座に await が返るので並列性能を損なわない)
6. **タブ反映 (逐次)**: 取得結果を `OpenThreadFromPrefetched(board, info, datResult)` に流す。HTTP は呼ばず、与えられた `DatFetchResult` から既存タブに差分 append または新規タブ作成 + Posts 反映 + idx.json 更新 + log mark 通知

**並列度の制御**:
- subject 取得 (Phase 3) と dat 取得 (Phase 5) で **同じ semaphore を共有** — 両者の通信本数の合計が `BatchConcurrency` (デフォ 6) を超えない
- ユーザがクリックして開く `OpenThreadAsync` には semaphore は介在しない (= バッチ実行中でもユーザ操作は待たされない、Phase 17 仕様)
- 画像系の HEAD / 取得 (`ImageMetaService` の独自 6 本制限、`WebView2` 内部スケジューラ) は対象外

**進捗表示** (StatusMessage):
- `お気に入りチェック: subject 取得 N/M`
- `お気に入りチェック: 更新スレ取得 N/M (完了 K/M)`
- `お気に入りチェック完了: 更新 N 件` または `お気に入りチェック完了: 新着なし`

**`OpenThreadFromPrefetched` ヘルパ**:
- `OpenThreadAsync` の前半 (タブ作成 / ViewMode / ScrollTarget / IsFavorited / State 初期化) と後半 (Posts append / idx 更新 / log mark) を再構成し、HTTP fetch 部分を取り除いたもの
- 既存タブがあれば差分のみ append、無ければ新規タブを `ThreadTabs.Add` + `SelectedThreadTab = tab` (= 現状の `OpenThreadAsync` と同じ挙動を踏襲)

### 7.11 全ログ機能 (Phase 18)

ローカルにキャッシュされている全 dat を 1 つのスレ一覧タブにまとめて表示する機能。お気に入りペイン上の「機能 → 全ログ」項目をクリックして起動。`MainViewModel.OpenAllLogsAsync` が実体。

**タブ識別**: `FavoritesFolderId == AllLogsTabId` (固定 sentinel `ffffffff-...-fffffffe`、`Guid.Empty` で衝突しないよう設計)。既存タブがあればアクティブ化のみ。

**Items ビルド** (`BuildAllLogsItems`):
1. `data/5ch.io/*/` および `data/bbspink.com/*/` 配下の各ディレクトリを enumerate
2. 各ディレクトリの `*.dat` を一覧化
3. ディレクトリ名から正規 `Board` を逆引き (`FindBoardByDirectory(rootDomain, dirName)`、サブドメイン違いを吸収)。`BoardCategories` に無ければ fallback Board (`Host = rootDomain`)
4. ディレクトリ内のローカル `_subject.txt` を読む (HTTP は呼ばない、`SubjectTxtClient.LoadFromDiskAsync`)
5. 各 dat ファイルについて:
   - subject.txt にエントリあり → そのまま `ThreadInfo` を採用、`BuildLogStates` で青/緑判定
   - subject.txt にエントリ無し (= 落ちた / subject 未取得) → dat の 1 行目から `name<>mail<>date_id<>body<>title` の 5 番目を SJIS で読んで title 抽出、`LogMarkState.Dropped` (茶) でマーク。post 数は `idx.json.LastFetchedPostCount` から復元 (無ければ 0)
6. お気に入り済キー集合 (`Favorites.CollectFavoriteThreadKeys()`) と突き合わせて `IsFavorited` を初期化

**リフレッシュ**: `RefreshThreadListTabAsync` 経由で `RefreshAllLogsTab` → `BuildAllLogsItems` を再実行 (HTTP なし、ディスク再 walk のみ)。タブ見出しは `📁 全ログ (N)`。

**JS 統合**: 通常のスレ一覧と同じ `ThreadListHtmlBuilder` を使うため、ログマーク・お気に入り増分通知・ソート・クリックイベントすべて動く。

### 7.12 スレ一覧のリアルタイム同期 (Phase 18 でリファクタ)

ログマーク (青/緑/茶) と お気に入り ★ の状態を、開いている全スレ一覧タブにリアルタイム反映する仕組み。

**同期対象**:
- 板タブ (Board 付き) — そのままのキー (host, dir, key) で行が一意
- 集約タブ (Board=null、お気に入り「板として開く」/ 全ログ) — 異なる板由来の行が混在するので、(host, dir, key) の **3 つ組** で行を一意特定

**ログマーク同期** (`NotifyThreadListLogMark(board, key, state)`):
- 全 `ThreadListTabs` を走査
- 板タブ: `Board.Host` / `DirectoryName` 一致でブロードキャスト
- 集約タブ: `ContainsThread(host, dir, key)` で該当行があるときのみ送信
- 各タブは `SetLogMark(host, dir, key, state)` で `LogMarkPatch` を発火 → JS は `tr[data-host=][data-dir=][data-key=]` で行を特定して `has-log` / `has-update` / `has-dropped` クラスを差し替える (= 再描画なし)

**お気に入り ★ 同期** (`RefreshFavoritedStateOfAllTabs`):
- ★ ボタン (`ThreadTab.IsFavorited`) と スレ一覧の行 ★ 背景を同時に更新
- 各 `ThreadListTab` で `SyncFavoritedFromKeySet(favKeys)` を呼ぶ:
  - 現在の `Items` と `favKeys` を突き合わせ、差分のみ `FavoritedChange(host, dir, key, isFavorited)` の patch を生成
  - `Items` の record も `with { IsFavorited = ... }` で同期更新 (= 次回の diff 計算が正しく動くよう C# 側のスナップショットを最新化)
  - 差分があれば `FavoritedUpdate` を発火 → JS は `is-favorited` クラスを toggle

**呼び出し点**:
- `ToggleThreadFavorite(tab)` — ★ ボタン押下
- `DeleteFavoriteEntry(vm)` — お気に入りペインからの削除 (フォルダ内の thread を含む)
- `AddBoardToFavorites(b)` — 板お気に入り追加 (将来用に統一して呼ぶ)
- `MoveFavoriteEntry(...)` — 移動 (状態は変わらないが念のため)
- `DeleteThreadLog(tab)` — dat 削除時にお気に入り登録があれば自動除外し、その後で sync を呼ぶ (Phase 18 の追加挙動)

**統合リフレッシュ** (`RefreshThreadListTabAsync(tab)`):
- 板タブ → `LoadThreadListAsync` (subject.txt 再取得)
- 全ログ → `RefreshAllLogsTab` (ディスク再 walk、HTTP なし)
- お気に入り集約 (Guid.Empty / フォルダ Id) → `BuildFavoritesAggregateItemsAsync` (subject.txt 再取得 + Items 再構築)
- `App.xaml.cs` の `thread_list.refresh` ショートカットハンドラはこの単一エントリを呼ぶだけ

### 7.13 「ここまで読んだ」帯 (Phase 19)

スレ本文の任意のレスの下に「ここまで読んだ」帯を出し、最深既読位置を可視化する機能。`idx.json.LastReadMarkPostNumber` で永続化し、dat 削除と同時に消える (= idx.json ごと削除)。

**判定基準** (JS 側):
- `#posts` の直下子要素 (= primary レス、`id="rN"` を持つ。tree モードの `.embedded` な子は除外される構造) のうち、`rect.bottom <= window.innerHeight` を満たす最大番号
- 短文レスは全表示で / 長文レスは下端を読み飛ばして抜けたタイミングで / 末尾レスはスレ最下端までスクロールしたときに条件成立
- 値は減少しない (C# 側 `MainViewModel.UpdateReadMark` で `Math.Max` 判定)

**通信フロー**:
1. JS: `scroll` イベント → 200ms debounce → `findDeepestReadMarkPostNumber()` で最深番号 → `readMarkSentMax` より大きければ C# に `{type:'readMark', postNumber}` を送信
2. C#: `MainWindow.HandleReadMark` → `MainViewModel.UpdateReadMark(board, key, n)` → 既存値より大きいときだけ `idx.json` 永続化 + タブの `ReadMarkPostNumber` を同期
3. `WebView2Helper.OnAppendBatchChanged`: `appendPosts` payload に `readMarkPostNumber` を同梱 (= タブを開いたときに JS が初期値を取り込む経路)

**帯のリアルタイム不動方針**:
- スクロール中は **C# 永続化のみ行い、帯位置 (`readMarkPostNumber`) は動かさない**
- `appendPosts` での値受信も初回の 1 度きり (`readMarkInitialized` フラグ) — リフレッシュで `appendPosts` が再発火しても帯は固定
- 結果: スレを開いている間は帯位置が変わらず、タブを閉じて再度開いたときに初めて新しい位置 (= 永続化された最新値) で表示される
- 設計判断の根拠: 帯がスクロールに合わせてリアルタイムに動くのは視覚的に煩わしい (実装直後にユーザフィードバックで確認)

**帯の DOM 配置**:
- `updateReadUpToBand()`: 既存帯を除去 → 対象レス (`document.getElementById('r' + readMarkPostNumber)`) の直後に挿入
- 対象レスが DOM に居ない場合 (NG であぼーん / view mode で primary でない 等): 番号 ≥ readMark の **最初の primary レス** の後ろにずらす
- 該当レスが無ければ帯は出ない (= 帯の対象レスがすべて消えている状況)

**リッチスクロールバー連動**:
- `track-readmark` トラックに薄紫 (`#9575cd`) の細い横線 1 本を描画
- 位置は帯 (or 対象レス本体) の `offsetTop + offsetHeight` を `scrollHeight` で割ったパーセント

**設定 ON/OFF**:
- `AppConfig.ShowReadMark` (デフォルト ON)、設定画面「スレッド」カテゴリのチェックボックス
- 設定 OFF にすると `<body>` に `no-read-mark` クラスが付き、CSS で帯と scrollbar マーカーが `display: none`
- **値の記録自体は止めない**ので、OFF にしてもバックグラウンドで `idx.json` は更新され、ON に戻すと過去最深位置から帯が出る
